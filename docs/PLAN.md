# Portal — 로컬 포트 관제탑 & 런처

> 브레인스토밍 수렴 문서 (2026-07-08)
> "지금 내 컴퓨터에서 뭐가 어느 포트에 떠 있는지 한눈에 보고, 버튼으로 끄고 켠다."

---

## 1. 이 앱이 정확히 뭘 하는가 (한 문장)

**포트를 쓰는 모든 로컬 프로세스를 메뉴바에서 한눈에 보여주고, 클릭 한 번으로 죽이거나(kill) 실행(start)하는 데스크톱 런처.**

브라우저 탭이 아니라 macOS 상단 메뉴바에 상주하는 독립 실행 앱(Tauri). 열자마자 아무 설정 없이 "지금 떠 있는 것들"이 보이고, 내가 등록한 프로젝트는 버튼으로 켤 수 있다.

## 2. 사람들이 얻는 결과 (Value)

| 이전 (Portal 없이) | 이후 (Portal로) |
|---|---|
| "3000 누가 물고 있지?" → `lsof -i:3000` 매번 검색 | 메뉴바 클릭 → 포트·앱·PID 즉시 목록 |
| 어제 띄운 서버가 안 죽어 오늘 앱이 3001로 튐 | 좀비/오래된 프로세스 눈에 띄고 버튼으로 kill |
| 앱마다 터미널 열어 `npm run dev` | 등록한 앱 옆 ▶ 버튼 한 번 |
| 뭐가 떠 있는지 감으로 기억 | 상단바 아이콘만 봐도 현황 파악 |

**핵심 가치 = "로컬 개발 프로세스의 가시성(visibility) + 원터치 제어"**

## 3. 핵심 결정 사항

| # | 결정 | 근거 |
|---|---|---|
| 1 | **정체성 = 관측(1순위) + 실행 런처** | 통증 1순위가 "한눈에 파악". 실행은 동급 MVP지만 관측이 앱의 얼굴 |
| 2 | **범위 = 누구나(범용)** | 내 factory 전용이 아니라 임의 프로젝트 등록 가능. OSS 배포 겨냥 |
| 3 | **대상 = 포트 쓰는 모든 프로세스** | `lsof -i -P`로 발견. 내 dev 서버뿐 아니라 시스템까지 전부 표시 |
| 4 | **MVP = 보기 + kill + 실행 셋 다** | 관측만으론 반쪽. 죽이고 켜는 것까지 있어야 "런처" |
| 5 | **kill 정책 = 그룹 구분(막진 않음)** | 내 dev / 시스템 / 기타로 그룹핑. 판단은 사용자. (확인 다이얼로그는 릴리스 직전 옵션) |
| 6 | **런타임 = Tauri** | ~10MB 경량, 프론트=React(익숙), 백엔드=Rust(신규 학습). OSS 배포에 유리 |
| 7 | **형태 = 메뉴바 상주형** | "끄고 켜기 쉽게"에 최적. 항상 손닿는 곳 |
| 8 | **이름 = Portal** | port + portal(관문). 기억 쉽고 의미 연결 |

## 4. 요구사항 (MoSCoW)

### Must (MVP)
- [ ] 메뉴바 트레이 아이콘 상주 + 클릭 시 팝오버 창
- [ ] 포트 점유 프로세스 자동 발견 (`lsof -i -P -n`) — PID / 포트 / 명령 / 프로세스명
- [ ] **그룹핑**: `내 dev 서버 / 시스템 / 기타`로 자동 분류해 표시
- [ ] kill 버튼 (프로세스 종료)
- [ ] 등록한 프로젝트 실행 버튼 (▶ start) — 지정 커맨드/cwd로 spawn
- [ ] 프로젝트 등록 UI (폴더 선택 + 실행 커맨드 + 포트)
- [ ] 상태 실시간 갱신 (polling 또는 이벤트)

### Should
- [ ] 좀비/오래된 프로세스 강조 (uptime 표시, 오래된 것 경고색)
- [ ] restart 버튼
- [ ] 포트 충돌 감지 ("3000은 이미 X가 씀")
- [ ] 로그 tail 보기 (실행한 프로세스의 stdout)
- [ ] 다크모드

### Could
- [ ] kill 확인 다이얼로그 (시스템/미확인 프로세스에만 마찰 추가)
- [ ] factory 폴더 자동 스캔 (설정 zero 온보딩)
- [ ] 프로젝트 그룹 실행 (`--group learn` 처럼 묶음 기동)
- [ ] 헬스체크 (등록한 healthPath 200 확인)
- [ ] 검색/필터

### Won't (이번엔 안 함)
- 원격/여러 머신 통합 (SaaS 영역 — 나중)
- 팀 동기화 / 클라우드
- 터널링(공유 URL) — 별도 제품 방향

## 5. 아키텍처 스케치

```
┌─ 메뉴바 트레이 (Tauri tray) ─────────────┐
│  [🔌] 클릭 → 팝오버 창                     │
└────────────┬──────────────────────────────┘
             │
┌────────────▼─── 프론트 (React + Vite) ────┐
│  - 프로세스 목록 렌더 / 그룹 UI            │
│  - kill / start / restart 버튼             │
│  - invoke("...") 로 Rust 호출              │
└────────────┬──────────────────────────────┘
             │ Tauri IPC (invoke / emit)
┌────────────▼─── 백엔드 (Rust commands) ───┐
│  scan_ports()   → lsof 파싱, 그룹 분류     │
│  kill_pid(pid)  → 프로세스 종료            │
│  start_app(cfg) → Command spawn (detached) │
│  watch loop     → 주기적 스캔 후 emit      │
│  registry       → 등록 프로젝트 저장(JSON) │
└────────────────────────────────────────────┘
```

### 데이터 흐름 (예: kill)
버튼 클릭 → `invoke("kill_pid", {pid})` → Rust가 `kill` 실행 → 성공 시 `scan_ports()` 재실행 → `emit("ports-updated")` → 프론트 리렌더

## 6. 핵심 기술 난관 (미리 아는 함정)

1. **프로세스 소유권 / detached spawn**
   - Portal이 앱을 spawn하면 Portal이 부모가 됨.
   - Portal 종료 시 자식도 죽을지(attached) 살려둘지(detached) 결정 필요 → **detached 권장** (오늘 겪은 좀비 문제의 반대편)
2. **"이미 떠 있는 것" vs "내가 띄운 것" 구분**
   - lsof로 발견한 PID와, Portal이 spawn해 추적 중인 PID를 매칭해야 restart/입양 가능
3. **lsof 파싱 & 그룹 분류 휴리스틱**
   - `내 dev`: cwd가 등록 프로젝트 하위 or node/next/vite 등 dev 시그니처
   - `시스템`: root 소유 or /System, /usr 경로
   - `기타`: 나머지
4. **권한**: lsof/kill은 대개 sudo 불필요(내 소유 프로세스). 시스템 프로세스 kill은 권한 부족으로 실패할 수 있음 → 에러 우아하게 처리
5. **Rust 첫 경험**: `std::process::Command`, `tauri::command`, 상태 공유(`State<Mutex<..>>`) 학습 필요

## 7. 구현 순서 (제안)

1. **Tauri 스캐폴딩** — 트레이 아이콘 + 빈 팝오버 창 뜨는 것 확인 (Hello World)
2. **scan_ports()** — lsof 파싱해서 프론트에 목록 렌더 (읽기 전용, 여기까지만 해도 유용)
3. **그룹 분류** — 내 dev / 시스템 / 기타
4. **kill_pid()** — 버튼으로 종료 + 자동 갱신
5. **프로젝트 등록 + start_app()** — 폴더 선택 → 실행
6. **watch loop / 실시간 갱신**
7. Should 항목 (좀비 강조, restart, 로그) 순차 추가
